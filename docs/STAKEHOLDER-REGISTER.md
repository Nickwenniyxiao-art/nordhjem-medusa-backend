# 干系人登记

## 干系人清单

| 干系人 | 类型 | 角色定位 | 主要诉求 | 影响等级 |
| --- | --- | --- | --- | --- |
| Owner (Nick) | 内部决策方 | 产品与商业最终负责人 | 按路线图交付、稳定上线、收入增长 | 高 |
| CTO (AI) | 内部执行方 | 技术架构与交付负责人 | 技术可行、交付效率、质量门禁达标 | 高 |
| 终端用户 | 外部客户 | 购买家具的顾客 | 浏览顺畅、支付安全、配送可追踪 | 高 |
| Stripe（支付） | 外部平台 | 支付网关与风控 | 支付成功率、合规与反欺诈 | 中 |
| Railway（部署） | 外部平台 | 应用托管与运行环境 | 服务可用性、部署稳定、成本可控 | 中 |

## 角色与职责

- **Owner（Nick）**：确认产品路线图、关键需求取舍、生产环境发布审批、预算策略确认。
- **CTO（AI）**：全权技术管理（架构设计、开发计划、质量控制、文档治理、发布准备）。
- **终端用户**：通过站点完成商品浏览、加购、结算、支付与订单跟踪；其反馈驱动迭代。
- **Stripe**：提供支付能力、交易状态回调、争议与退款能力。
- **Railway**：提供 Docker 运行环境、构建部署、运行时日志与可用性支持。

## 影响力与关注点

- **Owner**：高影响力；关注产品上线速度、营收增长、MVP 成本控制、品牌信誉。
- **终端用户**：高影响力（通过转化率和复购体现）；关注购物体验、支付安全、配送透明、售后保障。
- **CTO（AI）**：高影响力；关注系统稳定性、自动化程度、缺陷率、迭代效率。
- **Stripe**：中影响力；关注支付链路稳定、拒付率、合规接入（PCI/GDPR 相关）。
- **Railway**：中影响力；关注部署可重复性、资源限制下的性能与成本。

## 沟通机制

- **Owner ↔ CTO（AI）**：以 GitHub Issues/PR + Chat 为主，重大事项同步到 ROADMAP 与 TASK-REGISTRY。
- **CTO（AI）↔ 外部平台（Stripe/Railway）**：通过官方文档、控制台告警、状态页与工单处理。
- **终端用户 ↔ 团队**：通过网站客服入口、订单邮件、FAQ 反馈。
- **节奏**：
  - 每周一次迭代回顾（范围、风险、里程碑）；
  - 每次发布前执行发布清单确认；
  - 紧急事件按 Runbook 与 Incident 流程快速同步。
