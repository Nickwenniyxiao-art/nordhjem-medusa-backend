# PERFORMANCE-TEST

## 性能目标

- 首页 Core Web Vitals 目标：LCP < 2.5s、FID < 100ms、CLS < 0.1。
- Store API 与 Admin API 响应时间目标：< 500ms（P95）。
- 峰值承载目标：并发 100 用户下核心路径无明显降级（下单成功率不下降、错误率 < 1%）。
- 业务目标：在促销时段保持商品浏览、搜索、下单链路稳定，避免因性能抖动导致转化损失。

## 场景与负载模型

### 场景定义

1. **场景1：首页浏览（最高频）**
   - 用户访问 Next.js Store 首页，包含 Hero 区、分类入口、推荐商品卡片。
   - 涉及 SSR 首次渲染、Medusa Store API 聚合查询、图片静态资源加载。
2. **场景2：商品搜索**
   - 用户输入关键词进行搜索并切换筛选（分类、价格区间、库存）。
   - 涉及 API 查询、索引命中率、分页响应以及前端状态同步。
3. **场景3：下单结算（最关键）**
   - 从购物车进入结算，完成地址、配送、支付确认。
   - 涉及库存校验、运费计算、支付意图创建与确认（Stripe）。
4. **场景4：Admin 批量操作**
   - 运营在 Admin 后台批量更新价格/库存/上下架状态。
   - 涉及 Admin API 权限鉴权、批处理任务、数据库写入压力。

### 负载模型

- **常态流量**：50 QPS（白天稳定业务时段）。
- **峰值流量**：200 QPS（上新、折扣活动、社媒导流时段）。
- **并发目标**：100 活跃用户同时操作，重点关注下单路径错误率与支付回调稳定性。
- **压测建议工具**：k6（API）+ Lighthouse CI（前端）+ 数据库慢查询日志。

## 瓶颈分析与优化建议

### 已知瓶颈

1. **SSR 首次渲染耗时偏高**
   - 首页首屏依赖多接口聚合，冷启动或缓存失效时 LCP 波动明显。
2. **数据库 N+1 查询**
   - 商品列表及关联数据（variant、库存、价格）在部分查询路径存在重复访问。
3. **图片未优化**
   - 部分高分辨率产品图未做尺寸裁剪与现代格式压缩，影响首屏下载与 CLS 稳定性。

### 优化建议

1. **ISR 缓存策略**
   - 对首页/类目页实施 ISR + revalidate，降低 SSR 实时计算压力。
   - 高频榜单（热销/新品）采用短周期缓存，兼顾实时性与吞吐。
2. **查询优化**
   - 在 Medusa 数据访问层合并关联查询，减少 N+1。
   - 增加关键索引（如 product_handle、created_at、status），并持续审查 slow query。
3. **CDN 图片优化**
   - 启用 WebP/AVIF、按断点输出多尺寸图。
   - 对首屏图设置预加载策略，其余图片懒加载。
4. **性能守门机制**
   - 将 Lighthouse 预算、API P95 阈值纳入 CI 报告与发布前检查。
