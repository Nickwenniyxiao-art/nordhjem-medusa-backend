# ROLLBACK-PLAN

## 回滚触发条件

满足任一条件即触发回滚评估：

- 健康检查连续失败 > 3 次。
- 平台错误率 > 5% 且持续 10 分钟以上。
- API P95 响应时间 > 5s 且无快速缓解手段。
- 支付流程异常（支付成功但订单未落库、回调持续失败）。

## 回滚步骤

1. **Railway 回滚**

   - 在 Railway 控制台选择当前服务，执行 `revert to previous deployment`。

2. **Docker 回滚**

   - 拉取并重新部署上一稳定版本镜像（按已验证 tag）。

3. **数据库回滚**

   - 如涉及 schema 变更，执行对应 down migration。

4. **验证**

   - 重新跑健康检查与 smoke 测试，确认恢复成功。

## 数据一致性处理

回滚前必须核验以下事项：

- 是否包含**不可逆数据库迁移**（若不可逆，优先走数据修复方案）。
- 是否存在进行中的订单（避免重复扣款或订单状态错乱）。
- Stripe Webhook 队列是否已清空，避免旧回调污染回滚后数据。
- 必要时冻结下单入口，待一致性确认后再恢复流量。

## 复盘与改进

- 回滚后 24 小时内完成 Postmortem。
- 复盘文档归档到 `docs/POSTMORTEM/`。
- 根据复盘结果更新部署前检查清单与告警阈值。
- 若问题可自动化预防，必须追加 CI 或监控规则。
