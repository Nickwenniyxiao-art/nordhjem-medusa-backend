# 事故响应流程

## 严重级别定义

| 级别 | 描述 | 响应时间 | 示例 |
|------|------|---------|------|
| P0 | 服务完全不可用 | 15 分钟内 | 网站无法访问、数据库宕机 |
| P1 | 核心功能受损 | 30 分钟内 | 支付失败、订单无法创建 |
| P2 | 非核心功能异常 | 2 小时内 | 搜索变慢、邮件延迟 |
| P3 | 轻微问题 | 24 小时内 | UI 错位、非关键日志报错 |

## 响应流程

### 1. 发现 (Detection)
- Uptime Kuma 告警
- Smoke Test 失败
- Sentry 错误激增
- 用户反馈

### 2. 评估 (Assessment)
- 确认影响范围
- 评定严重级别
- 通知相关人员（Telegram）

### 3. 响应 (Response)
- P0/P1: 立即回滚到上一个稳定版本
- P2: 创建 hotfix 分支修复
- P3: 纳入正常开发流程

### 4. 修复 (Resolution)
- 实施修复方案
- 验证修复效果
- 更新状态

### 5. 复盘 (Postmortem)
- 48 小时内完成 Postmortem
- 使用标准模板
- 提取改进项并创建 Issue
