## 责任边界 — [详细说明该章节应包含的内容、格式要求和最佳实践示例]

本章节定义服务责任边界，明确“谁拥有数据、谁执行业务规则、谁对外暴露能力”。
应识别跨服务共享逻辑并制定归属原则，避免重复实现与耦合扩散。
建议以 DDD 术语（聚合、上下文）描述边界，并配合常见冲突案例。
需记录边界变更审批流程与影响评估要求。
最佳实践：对每条边界增加可验证条目，如“该服务是否直接写入他域主数据”。
边界检查清单：
- [ ] 单一数据主责明确
- [ ] 无跨域数据库直连
- [ ] 公共能力经 API 暴露
- [ ] 例外场景有书面批准

## 生命周期管理 — [详细说明该章节应包含的内容、格式要求和最佳实践示例]

本章节说明服务从立项到退役的全生命周期管理，包括准入、发布、变更、退役。
应定义每个阶段必须完成的治理动作，如威胁建模、容量评估、SLO 验证与成本审查。
建议引入生命周期状态机（Proposal/Active/Deprecated/Retired）并设置进入条件。
需明确退役时的数据迁移、消费者通知与接口冻结窗口。
最佳实践：用 checklist 驱动发布门禁，避免“技术完成但治理未完成”的上线风险。
