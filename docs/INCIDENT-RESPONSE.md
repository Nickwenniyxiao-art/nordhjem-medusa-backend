# NordHjem 事件响应流程

> 定义：什么时候做什么、谁来做、做完之后要什么
> 最后更新：2026-03-17

---

## 严重级别定义

| 级别 | 名称 | 定义 | 举例 |
|------|------|------|------|
| SEV1 | 全面宕机 | 生产环境完全不可用，所有用户受影响 | 后端 Health Check 失败、数据库崩溃、服务器宕机 |
| SEV2 | 部分降级 | 核心功能受影响，但服务部分可用 | 支付无法处理、搜索失败、部分 API 超时 |
| SEV3 | 小问题 | 非核心功能异常，多数用户不受影响 | 某个管理后台页面报错、非关键 API 返回错误 |

---

## 响应时间 SLA

| 级别 | 响应时间 | 修复目标 | 通知 Owner |
|------|----------|----------|-----------|
| SEV1 | < 30 分钟 | < 2 小时 | 立即通知 |
| SEV2 | < 2 小时 | < 8 小时 | 尽快通知 |
| SEV3 | < 24 小时 | < 3 天 | 下次同步时提及 |

> "响应时间"指从发现问题到开始排查的时间，不是修复完成的时间。

---

## 事件响应流程

### Phase 1：发现（Detection）
事件可能通过以下渠道发现：
- **自动告警**：alert-on-failure.yml 创建的 GitHub Issue（标签 `ci-alert`）
- **监控告警**：Uptime Kuma 发送的通知
- **用户报告**：Owner 或用户反馈
- **巡检发现**：weekly-inspection 或 smoke-test 发现

### Phase 2：分级（Triage）
1. 评估影响范围 → 确定 SEV 级别
2. 创建 GitHub Issue（使用 `incident.yml` 模板）
3. 设置标签：`type: incident` + `SEV1`/`SEV2`/`SEV3`
4. 设置 Project Board 字段：Status=In Progress, Priority=P0/P1/P2

### Phase 3：响应（Response）
1. **SEV1**：
   - 立即通知 Owner
   - 参考 RUNBOOK.md 执行诊断和修复
   - 如果 5 分钟内不能定位原因 → 执行回滚
   - 每 30 分钟更新 Issue 进展
2. **SEV2**：
   - 通知 Owner
   - 参考 RUNBOOK.md 排查
   - 如果有降级方案（如关闭有问题的功能），先执行降级
3. **SEV3**：
   - 记录到 Issue
   - 安排在下个迭代修复

### Phase 4：恢复（Recovery）
1. 确认服务恢复正常（Health Check 通过）
2. 验证用户功能可用
3. 更新 Issue 状态为 Done
4. 如果执行了回滚 → 创建修复任务 → 走正常 CI/CD 流程重新部署

### Phase 5：事后分析（Postmortem）

**SEV1 和 SEV2 事件必须写 Postmortem。** SEV3 可选。

Postmortem 文件路径：`docs/postmortem/YYYY-MM-DD-<简短描述>.md`

Postmortem 模板：
```
# Postmortem: <事件标题>

**日期**: YYYY-MM-DD
**级别**: SEV1 / SEV2
**持续时间**: X 小时 Y 分钟
**影响**: 描述受影响的功能和用户范围
**关联 Issue**: #xxx

## 时间线
| 时间 | 事件 |
|------|------|
| HH:MM | 发现问题 |
| HH:MM | 开始排查 |
| HH:MM | 执行修复/回滚 |
| HH:MM | 服务恢复 |

## 根本原因
详细描述导致问题的根本原因。

## 修复措施
描述实施了什么修复。

## 经验教训
### 做得好的
- ...
### 需要改进的
- ...

## 后续行动
| 行动项 | 负责人 | 截止日期 | Issue |
|--------|--------|----------|-------|
| ... | ... | ... | #xxx |
```

## 快速决策树

```
服务异常
  ├─ 生产完全不可用？
  │   ├─ 是 → SEV1 → 立即通知 Owner → 5分钟定位 → 否则回滚
  │   └─ 否 → 核心功能受影响？
  │       ├─ 是 → SEV2 → 通知 Owner → 排查或降级
  │       └─ 否 → SEV3 → 记录 Issue → 下个迭代修复
  └─ 修复/回滚完成后
      └─ SEV1/SEV2 → 必须写 Postmortem
```
