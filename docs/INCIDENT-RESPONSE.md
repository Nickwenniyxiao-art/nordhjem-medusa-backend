# INCIDENT-RESPONSE

## 事故分级

- **P0**：全站不可用 / 数据泄露。
- **P1**：核心功能（支付/下单）不可用。
- **P2**：非核心功能异常 / 性能降级。
- **P3**：UI 问题 / 非紧急 bug。

## 响应流程

1. **发现（0-5 分钟）**

   - 来源：监控告警、用户反馈、业务异常指标。
   - 负责人：值班 CTO。

2. **确认（5-10 分钟）**

   - 复现问题、确认影响范围与受影响模块。
   - 负责人：CTO。

3. **分级（10-15 分钟）**

   - 按 P0-P3 规则定级并更新 incident 记录。
   - 负责人：CTO，P0/P1 同步 Owner。

4. **响应（按级别 SLA）**

   - 先止损（限流、降级、回滚）再定位根因。
   - 负责人：CTO 主导。

5. **修复**

   - 发布 hotfix 或执行回滚，确保核心流程恢复。

6. **验证**

   - 复跑健康检查、smoke 测试、关键业务路径。

7. **复盘**

   - 形成正式 postmortem 并跟踪改进项闭环。

## 沟通机制

- **P0/P1**：立即通知 Owner（GitHub Issue + 标签 `priority: P0` / `priority: P1`）。
- **P2/P3**：在下次同步会议汇报，必要时创建低优先级修复任务。
- 所有事件需保留完整时间线与沟通记录，便于审计与复盘。

## 事后复盘

- 事故结束后 **48 小时内**完成 Blameless Postmortem。
- 使用 `docs/POSTMORTEM/` 模板归档。
- 必须包含：**时间线、根因、影响范围、改进措施**。
- 改进措施需映射到具体 Action/Issue 并跟踪到完成。
