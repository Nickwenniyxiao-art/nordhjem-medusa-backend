# 新项目文档评估模板

## 评估概述
本模板用于新项目启动阶段，帮助团队快速判断“哪些文档必须先准备、哪些文档可按需补充”。
使用场景：立项会后、需求冻结前、进入开发前的文档门禁检查。
目标：在项目初期形成一份可执行、可追踪、可审计的文档清单，避免后续返工。

## 文档匹配算法
以下规则用于从 `DOC-LIBRARY.json` 自动生成“建议文档清单”。

### 输入
1. 已确认的主类型（单选）。
2. 已勾选的项目特性（多选）。
3. 文档库 `DOC-LIBRARY.json`（包含每个文档的 `applicable_to` 与 `default_required` 字段）。

### 匹配逻辑（必须按顺序执行）
1. **筛选基础集合**：读取 `DOC-LIBRARY.json`，获取全部文档条目。
2. **加入通用文档**：`applicable_to: ["all"]` 的文档默认包含。
3. **加入类型专项文档**：`applicable_to` 包含“主类型”或“项目特性”关键词的文档一并包含。
4. **标记必须项**：`default_required: true` 的文档标记为“必须”。
5. **生成初稿清单**：输出“文档名称 + 适用原因 + 是否必须”。
6. **Owner 复核并调整**：Owner 可手动删除不适用项，也可新增项目特有文档。

### 输出建议格式
- 文档名称
- 来源规则（all / 类型匹配 / 特性匹配 / Owner 追加）
- 必须性（必须 / 建议）
- 备注（如：由合规要求触发）

> 示例（NordHjem）：
> - `ARCHITECTURE.md`（来源：all，必须）
> - `SECURITY-POLICY.md`（来源：all，必须）
> - `API-SPEC.md`（来源：有外部 API，建议）
> - `DATA-COMPLIANCE.md`（来源：GDPR，Owner 追加为必须）

## 文档清单审核
Owner 按以下步骤审核自动匹配结果：

1. 核对每个文档是否与当前阶段相关（立项/开发/上线）。
2. 对“必须”文档确认责任人与完成时间。
3. 删除明显不适用文档，并写明删除原因（便于审计）。
4. 追加项目特有文档（如第三方接入规范、行业合规说明）。
5. 输出最终版清单并冻结版本号（例如 v1.0）。

建议使用表格记录：

| 文档名 | 是否必须 | 责任人 | 截止日期 | 状态 | 备注 |
| --- | --- | --- | --- | --- | --- |
| 例：ARCHITECTURE.md | 是 | @owner | 2026-03-20 | 进行中 | 初版架构图待补充 |

- 已确认阶段门禁要求（未达标不可进入下一阶段）

签字信息：

## 后续步骤
签字确认后，按以下流程执行：

1. 根据最终清单生成 `DOC-REGISTRY.json`。
2. 为每个文档写入状态字段（未开始/进行中/已完成/阻塞）。
3. 将 `DOC-REGISTRY.json` 提交到仓库，并关联 Issue/PR。
4. 在里程碑评审（如 Phase Gate）中使用 `DOC-REGISTRY.json` 作为检查依据。
5. 每次范围变更（新增模块、合规要求变化）后，重新执行本模板并更新登记表。
