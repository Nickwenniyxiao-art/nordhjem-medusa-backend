## 监控规则 — [详细说明该章节应包含的内容、格式要求和最佳实践示例]
监控规则应覆盖资金流异常、行为异常、关联网络异常与地理异常，区分实时规则与离线模型规则。  
规则定义应包含：规则 ID、业务场景、触发条件、阈值、观测窗口、排除条件、风险评分、处置动作。  
建议采用“规则分级”管理：L1（提示）、L2（需调查）、L3（高危立即管控），并与工单优先级联动。  
需建立规则有效性评估机制，通过命中率、转案件率、误报率等指标按 [周期] 进行调优。  
高风险规则上线前应执行回放测试（历史样本）并由合规、风控、业务三方签字审批。

## 可疑交易识别 — [详细说明该章节应包含的内容、格式要求和最佳实践示例]
可疑交易识别应结合规则告警、客户画像、交易上下文与外部情报，形成“机器识别 + 人工判断”双轨机制。  
分析员应依据 [调查模板] 记录资金来源、资金去向、受益关系、交易目的、与历史行为偏离程度。  
建议设置案件分层：观察（Watch）、调查（Investigate）、可疑（Suspicious）、确认上报（Reportable）。  
当命中高风险名单或出现明显洗钱特征（分层转移、循环转账、走账空转）时，应触发即时管控措施。  
每起可疑案件应形成标准化证据包，确保后续监管问询和司法协作可快速响应。  

**识别检查清单（模板）**
- [ ] 是否存在无合理商业目的的复杂交易路径。
- [ ] 是否存在短时间资金集中进出与余额归零特征。
- [ ] 是否命中制裁名单、PEP 或负面舆情。
- [ ] 是否存在关联账户协同行为与设备重合。
- [ ] 调查结论与证据是否可复核、可复现。

## 报送流程 — [详细说明该章节应包含的内容、格式要求和最佳实践示例]
报送流程需定义从“案件认定 → 内部复核 → 法务/合规审批 → 监管报送 → 回执归档”的全链路时限与责任人。  
应明确报送触发条件、报送时点（T+N）、报送渠道（监管平台/API/加密邮件）及补正流程。  
报送材料至少包含：客户身份信息、交易明细、可疑特征说明、调查过程、处置动作与附件证据。  
建议设置四眼原则（经办 + 复核）和报送质量校验（字段完整性、逻辑一致性、附件可读性）。  
报送后应记录回执编号 [回执编号] 与状态跟踪，若被退回需在 [时限] 内完成更正并再次提交。

## 审计追踪 — [详细说明该章节应包含的内容、格式要求和最佳实践示例]
审计追踪应覆盖规则生命周期、告警处理链路、人工决策依据、报送记录与数据访问日志。  
关键日志字段建议包括：事件 ID、规则版本、操作者、时间戳、变更前后值、审批节点、证据引用。  
系统应提供不可篡改审计能力（如哈希签名、WORM 存储）并支持按案件一键导出审计包。  
每季度执行一次 AML 审计抽查，验证规则有效性、流程执行一致性、权限最小化与数据保留合规。  
对于重大事件，应在 [时限] 内完成根因分析（RCA）与控制改进计划，跟踪整改完成率。  

**审计要点清单（模板）**
- [ ] 规则新增/调整/下线均有审批和验证证据。
- [ ] 告警处置存在完整的时间线和责任链。
- [ ] 报送记录与监管回执可双向对应。
- [ ] 审计日志支持按客户、规则、案件多维检索。
- [ ] 抽查发现问题已形成 CAPA（纠正预防措施）。
