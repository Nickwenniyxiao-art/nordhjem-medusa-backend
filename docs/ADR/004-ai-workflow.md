# ADR-004: AI 工作流设计

## 状态

已采纳

## 日期

2026-03-13

## 背景

NordHjem 的主要开发力量是 AI（OpenAI Codex），人工主要负责架构决策和最终审核。这带来两个核心问题：

1. **CI 失败后如何处理**：Codex 生成的代码有时会引入 Lint 错误或测试失败，需要有机制自动修复，否则开发者需要频繁手动介入
2. **代码质量如何保障**：AI 生成的代码量大、速度快，人工 Review 无法跟上节奏，需要 AI 来辅助审查 AI

当时的痛点：

- CI 失败后，Codex 有时会反复提交同样的错误修复，进入"死循环"
- 没有系统性的代码审查，低质量代码直接进入 develop 分支

## 方案对比

### CI 失败修复机制

| 方案 | 优点 | 缺点 |
|------|------|------|
| 人工修复 | 修复质量高、可理解业务上下文 | 无法 7×24 响应；高频 CI 失败时人工成本巨大 |
| Codex Auto-fix（无限重试） | 全自动、无需人工 | 可能陷入死循环；修复质量不稳定；无 scope 限制可能引入更大改动 |
| Codex Auto-fix（有重试上限 + scope 限制） | 自动化同时有安全边界；死循环风险可控 | 超出重试上限后仍需人工介入 |

### 代码审查机制

| 方案 | 优点 | 缺点 |
|------|------|------|
| 单一 AI 审查（GPT-4o-mini） | 成本低、响应快 | 单一模型视角，可能有系统性盲区 |
| AI 三审制（GPT-4o-mini + Claude + 架构专项审查） | 多角度覆盖，审查更全面 | API 成本高；三个审查串联增加 PR 等待时间 |
| AI 三审制分阶段启用 | 兼顾成本和质量；先启用快速审查，逐步增加覆盖 | 初期覆盖不完整 |

## 决策

采用 **Codex Auto-fix（受控重试）+ AI Review 三审制（分阶段启用）**：

### Codex Auto-fix 规则

- **触发条件**：CI 流水线失败（Lint/Test/TypeScript 错误）
- **重试上限**：最多 3 次，超过后自动创建 Issue 并通知人工
- **Scope 限制**：Auto-fix 只允许修改导致 CI 失败的具体文件，不得新增功能或重构无关代码
- **安装命令**：使用 `npm install --legacy-peer-deps`（非 `npm ci`），避免依赖冲突导致 Auto-fix 自身失败进而死循环（见 ADR-007）
- **退出条件**：连续 3 次修复后 CI 仍失败，停止 Auto-fix，标记 PR 需要人工介入

### AI Review 三审制

| 审查层 | 模型 | 状态 | 审查重点 |
|--------|------|------|----------|
| 快速审查 | GPT-4o-mini | **已启用** | 代码规范、明显错误、安全漏洞、性能反模式 |
| 深度审查 | Claude Sonnet | 待启用 | 业务逻辑正确性、边界条件、错误处理 |
| 架构审查 | GPT-4o / Claude Opus | 待启用 | 架构合理性、模块耦合、可扩展性 |

PR 门禁：GPT-4o-mini 审查通过为**必要条件**（Required Check），其余两层为建议性反馈。

## 后果

### 正面影响

- Codex Auto-fix 显著减少人工干预频率，CI 失败通常在 1-2 次 Auto-fix 内自愈
- 重试上限 + Scope 限制有效防止了死循环和不可控的代码膨胀
- GPT-4o-mini Review 已覆盖基础质量门禁，拦截了明显的 Lint 问题和安全风险
- 分阶段启用策略控制了 API 成本，随项目成熟逐步提升审查深度

### 负面影响 / 需关注的风险

- Auto-fix 修复的代码质量不稳定，可能产生"能跑但不优雅"的代码，需定期人工审计
- 超出 3 次重试上限后，人工需要理解 AI 的修复历史才能继续，上下文切换成本高
- Claude 和架构审查层尚未启用，当前存在审查盲区（业务逻辑层面）
- AI Review 依赖 OpenAI API，API 限流或故障会阻塞所有 PR 合并
