# ADR-001: 技术栈选型

## 状态

已采纳

## 日期

2025-01-01

## 背景

NordHjem 是一个定位北欧极简主义风格的家居电商独立站。项目立项时需要确定完整的技术栈。核心诉求包括：

- **电商能力**：商品管理、购物车、结账、订单、支付集成
- **SEO 友好**：家居品类依赖自然搜索流量，需要服务端渲染
- **高度可定制**：北欧极简设计风格对前端 UI 有强烈的定制需求，不希望被主题系统限制
- **长期可控**：不希望过度依赖 SaaS 平台，保留数据和逻辑的完全所有权
- **团队规模**：早期小团队（含 AI 辅助开发），需要控制技术复杂度

## 方案对比

### 电商平台

| 方案 | 优点 | 缺点 |
|------|------|------|
| Shopify | 开箱即用、生态完善、运维成本低 | 月费高、定制受限（主题系统）、数据不完全自控、Headless 额外收费 |
| WooCommerce | WordPress 生态丰富、插件多 | 基于 PHP，与前端技术栈割裂；性能较差；安全补丁频繁 |
| Medusa.js v2（自建） | 开源免费、完全 Headless、Node.js 生态、高度可定制、数据自主 | 需要自行运维、相对年轻、社区规模不及 Shopify |

### 前端框架

| 方案 | 优点 | 缺点 |
|------|------|------|
| React / CRA | 简单、熟悉 | 纯 CSR，SEO 不友好；CRA 已停止维护 |
| Next.js 15 | SSR/SSG/ISR 灵活、SEO 友好、App Router 成熟、Vercel 生态 | 学习曲线略高；服务端组件模型需要适应 |
| Remix | 全栈友好、表单处理好 | 社区相对较小；与 Medusa 的集成案例少 |

### 数据库

| 方案 | 优点 | 缺点 |
|------|------|------|
| MySQL | 广泛使用、运维经验丰富 | Medusa v2 官方不推荐；JSON 支持较弱 |
| PostgreSQL | Medusa v2 官方推荐、JSON 支持强、事务严格、扩展丰富 | 相对 MySQL 内存占用略高 |
| SQLite | 零运维、适合开发环境 | 不适合生产多并发场景 |

## 决策

选择 **Medusa.js v2 + Next.js 15 + PostgreSQL + Redis** 组合：

- **Medusa.js v2**：开源 Headless 电商后端，完全控制数据和业务逻辑，Node.js 生态与前端一致，避免 SaaS 平台锁定
- **Next.js 15**：App Router 支持 SSR/SSG/ISR，SEO 友好，与 React 生态无缝衔接，社区最活跃
- **PostgreSQL**：Medusa v2 官方支持和推荐，JSONB 支持产品属性的灵活存储
- **Redis**：用于 Session 缓存、队列（Medusa 事件系统）和限流

## 后果

### 正面影响

- 完全掌控数据和业务逻辑，无 SaaS 平台依赖
- Next.js SSR 保证 SEO 竞争力
- Node.js 全栈统一语言，AI 辅助开发效率高
- Medusa v2 的 Module 架构便于后续扩展（多仓、多币种、B2B）

### 负面影响 / 需关注的风险

- 自建电商后端需要承担更多运维职责（数据库备份、版本升级）
- Medusa v2 相对年轻，部分功能仍在迭代，升级可能有 Breaking Change
- 相比 Shopify 开箱即用，前期开发工作量更大
- 团队需要同时维护前后端两套 Node.js 项目
