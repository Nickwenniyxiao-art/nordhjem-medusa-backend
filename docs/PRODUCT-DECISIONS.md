# PRODUCT-DECISIONS

## 决策背景

产品决策日志（Product Decisions Log）用于沉淀与业务方向、产品策略、平台选择相关的关键决策，确保团队在后续实施、审计与复盘时可以追溯“为什么这样做”。

- **谁来写**：默认由 Owner、AI CTO、模块负责人在关键决策达成后补录。
- **何时写**：发生架构级、平台级、交付模式级、市场策略级决策时，在对应 Issue/PR/ADR 建立后同步登记。
- **如何使用**：本日志作为 ADR 的上游业务语境补充，强调“产品与交付选择”；技术细节下钻到 ADR 文档。

### 历史决策记录

### PD-001: 选择 Medusa v2 作为电商引擎

- **日期**: 2026-03-13
- **决策者**: Owner
- **状态**: 接受
- **背景**: 需要一个支持可扩展模块化能力、可自建后端、便于与自定义业务流程集成的电商引擎。
- **选项**: Medusa v2 / Shopify(托管) / Saleor / 自研后端。
- **决策**: 采用 Medusa.js v2 作为 NordHjem 后端核心电商引擎。
- **理由**: Medusa v2 在可扩展性、TypeScript 生态、模块化定制能力方面更匹配当前团队与业务节奏，并支持后续自定义工作流。
- **影响**: 后端开发以 Medusa v2 模块范式为中心，相关 API、数据模型与插件集成策略围绕该技术栈展开。
- **关联**: docs/ADR/001-tech-stack.md, docs/ARCHITECTURE.md。

### PD-002: 选择 Railway 作为部署平台

- **日期**: 2026-03-13
- **决策者**: Owner
- **状态**: 接受
- **背景**: 需要快速建立可持续部署环境，并降低早期 DevOps 维护复杂度。
- **选项**: Railway / Render / 自建 VPS + Docker / Kubernetes。
- **决策**: 在当前阶段选用 Railway 作为主要部署平台。
- **理由**: Railway 对 Node.js + PostgreSQL 场景集成路径短，上线速度快，适合当前迭代周期和资源配置。
- **影响**: CI/CD、环境变量管理、部署排障流程围绕 Railway 能力设计，后续如迁移需评估供应商锁定风险。
- **关联**: docs/ADR/005-deployment-arch.md, docs/ADR/ADR-002-deployment-strategy.md。

### PD-003: 采用 AI CTO + Codex 全 AI 驱动开发模式

- **日期**: 2026-03-14
- **决策者**: Owner + AI CTO
- **状态**: 接受
- **背景**: 需要在有限人力下保持高频交付、强约束治理与可审计执行过程。
- **选项**: 传统人工主导 / AI 辅助人工主导 / AI CTO + Codex 全 AI 驱动。
- **决策**: 采用 AI CTO + Codex 协同执行模式，并建立 CI Gate 与 EGP 追溯机制。
- **理由**: 可在保持规范化（Issue/PR/ROADMAP/EGP）前提下提高执行吞吐量，并实现过程透明化。
- **影响**: 研发流程对文档与门禁依赖提高，需持续维护任务注册表、PR 元数据和质量审查规则。
- **关联**: docs/ADR/004-ai-workflow.md, docs/ADR/011-execution-guarantee.md, docs/ROADMAP.md。

### PD-004: 三环境部署策略（test/staging/production）

- **日期**: 2026-03-13
- **决策者**: Owner
- **状态**: 接受
- **背景**: 需要在保证交付速度的同时降低直接变更生产环境的风险。
- **选项**: 单环境直发 / 双环境（staging+production）/ 三环境（test+staging+production）。
- **决策**: 采用 test / staging / production 三环境分层发布策略。
- **理由**: 可先在 test 验证功能，再在 staging 验证集成与回归，最终进入 production，风险隔离更清晰。
- **影响**: CI/CD、数据库与配置管理需支持多环境隔离，发布流程增加但整体稳定性提升。
- **关联**: docs/ADR/ADR-002-deployment-strategy.md, docs/ROADMAP.md, AGENTS.md。

### PD-005: 面向北欧市场，支持 DKK/EUR/GBP 多币种

- **日期**: 2026-03-13
- **决策者**: Owner + 产品负责人
- **状态**: 接受
- **背景**: 目标市场覆盖北欧与欧洲跨境用户，单币种策略无法满足本地化交易需求。
- **选项**: 仅 DKK / DKK+EUR / DKK+EUR+GBP。
- **决策**: 第一阶段支持 DKK、EUR、GBP 三币种。
- **理由**: 在复杂度可控前提下覆盖核心目标市场，平衡本地化体验与结算实施成本。
- **影响**: 价格展示、税费、支付配置与报表需要按币种处理，并在后续扩展 i18n 与地区化策略。
- **关联**: docs/ROADMAP.md, docs/PROJECT.md。

## 影响评估

- 以上决策共同确定了项目的“平台 + 交付模式 + 发布策略 + 市场范围”基线。
- 对工程侧影响：需要持续维护 ADR、ROADMAP、TASK-REGISTRY 与 CI Gate 一致性。
- 对产品侧影响：功能优先级需兼顾多币种与多环境验证成本，避免超前复杂化。
- 对运营侧影响：部署平台与三环境策略提升稳定性，但要求更严格的变更与监控流程。
