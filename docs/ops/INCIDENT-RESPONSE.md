# 事件响应流程

## 目标与适用范围

本文档定义 NordHjem 后端生产环境事件响应标准流程，覆盖事件严重等级、响应步骤、角色分工与沟通模板。

适用范围：
- 生产环境可用性与稳定性故障
- 数据安全与隐私相关事件
- 核心业务链路（登录、下单、支付、订单处理）异常

## 严重等级定义

| 等级 | 描述 | 响应时间 | 示例 |
|------|------|----------|------|
| P0 - 紧急 | 生产完全不可用 | 15分钟 | 网站宕机、数据泄露 |
| P1 - 高 | 核心功能受损 | 1小时 | 支付失败、订单无法创建 |
| P2 - 中 | 非核心功能受损 | 4小时 | 搜索降级、邮件延迟 |
| P3 - 低 | 体验问题 | 下个工作日 | UI 错位、性能轻微下降 |

## 响应流程

1. **检测与报告**
   - 来源：监控告警、客服反馈、业务方上报、值班巡检。
   - 要求：首次发现后立即在事件频道登记（时间、现象、影响范围）。

2. **初始评估与分级**
   - 判断影响范围：用户数、业务损失、数据风险、合规风险。
   - 按 P0-P3 完成分级并确认 SLA 响应时限。

3. **组建响应团队**
   - 指定事件指挥官（IC）。
   - 拉齐通信协调与技术负责人，必要时引入产品、客服、法务。

4. **调查与修复**
   - 进行根因定位（日志、指标、链路、近期变更）。
   - 采取临时止血方案（降级、回滚、限流）并推进永久修复。

5. **验证与恢复**
   - 通过关键监控指标、业务冒烟测试验证修复有效。
   - 明确恢复时间、剩余风险与观察窗口。

6. **事后复盘 (Postmortem)**
   - 在 48 小时内完成复盘草案。
   - 输出根因、影响评估、改进动作与责任人/截止时间。

## 角色分工

- **事件指挥官 (IC)**
  - 负责事件分级、优先级决策、节奏推进。
  - 对整体恢复结果负责，决定升级或降级响应。

- **通信协调**
  - 维护内外部信息同步节奏。
  - 负责状态更新发布，保证单一信息出口。

- **技术负责人**
  - 负责技术排障方案制定与执行。
  - 协调开发、运维、数据库等资源完成修复与验证。

## 沟通模板

### 内部通知模板

```text
[Incident][<等级>] <事件标题>
- 发现时间: <YYYY-MM-DD HH:mm TZ>
- 当前状态: Investigating / Mitigated / Monitoring / Resolved
- 影响范围: <服务/用户/功能>
- 当前措施: <已执行动作>
- 下一次更新: <YYYY-MM-DD HH:mm TZ>
- IC: <姓名>
- 技术负责人: <姓名>
- 通信协调: <姓名>
```

### 对外状态更新模板

```text
我们正在处理一起影响 <功能/服务> 的故障。
事件开始时间：<YYYY-MM-DD HH:mm TZ>
当前影响：<用户可感知影响>
当前进展：<排查/修复状态>
下一次更新时间：<YYYY-MM-DD HH:mm TZ>
对给您带来的不便，我们深表歉意。
```

## 复盘与持续改进要求

- 所有 P0/P1 事件必须产出 Postmortem 文档并关联改进任务。
- P2/P3 事件按周汇总，评估是否需要升级治理项。
- 改进任务需纳入 ROADMAP 与 TASK-REGISTRY 跟踪，直至关闭。
