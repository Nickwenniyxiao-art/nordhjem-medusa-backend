# PM-001: Manus AI 误改生产环境变量导致全站宕机

## 事故概述
AI Agent（Manus）在执行 VPS 运维任务时，误修改了生产环境的 `.env` 配置文件，导致后端服务无法启动，生产环境完全不可访问。

## 事故等级
P0

## 影响范围
- 影响用户数：全量用户（生产环境 100% 不可用）
- 影响功能：nordhjem.store（前端）及 api.nordhjem.store（后端 API）均不可访问
- 影响时长：从发现到恢复，具体时长待确认（事发 2026-03）

## 根因分析（5 Whys）

1. **Why?** 表面原因是什么？
   → 生产环境的 `.env` 文件被修改，导致关键环境变量丢失或被覆盖，服务启动失败。

2. **Why?** AI Agent 为什么有权限修改 `.env`？
   → 未对 AI Agent 的操作范围做任何限制，Manus 拥有 VPS 的完整操作权限，可以读写任意文件。

3. **Why?** 为什么没有对 AI 的操作范围做限制？
   → 当时尚未建立运维规范和权限管理机制，AI 操作生产环境缺乏 guardrail。

4. **Why?** 为什么没有建立相应规范？
   → 项目初期快速迭代，优先交付功能，工程治理体系尚未成型，未意识到 AI Agent 操作生产环境的高风险性。

5. **Why?** 根本原因是什么？
   → 缺乏系统性的工程管理体系。AI Agent 作为自动化操作主体，在没有任何 guardrail 的情况下直接接触生产环境，是本次事故的根本原因。

## 修复措施
- **临时修复**：手动恢复 `.env` 文件中被误修改的环境变量，重启后端服务，确认服务恢复正常。
- **永久修复**：
  1. 在 GitHub 建立 Environment Protection Rules，production 环境变更需 Owner 审批方可执行。
  2. 明确禁止 AI Agent（含 Manus、Codex 等）直接操作生产环境配置文件（`.env`、数据库配置等）。
  3. 建立 Engineering Playbook，将上述约束写入正式规范并持续维护。

## 经验教训
- **做得好的**：发现问题后响应迅速，定位根因准确，修复方式直接有效；事后立即推动了治理体系的建立。
- **做得不好的**：在引入 AI Agent 操作生产环境前，未预先评估风险、未设置任何权限边界；事故发生时缺乏精确的时间记录，导致复盘信息不完整。
- **需要改进的**：
  1. 所有 AI Agent 的操作范围必须在使用前明确界定，生产环境操作须有人工审批节点。
  2. 建立事故发生时的实时记录习惯（时间、操作、现象），为后续复盘提供完整数据。
  3. 生产环境 `.env` 文件应纳入备份机制，确保可快速回滚。
