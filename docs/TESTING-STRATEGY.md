# TESTING-STRATEGY

## 1. 概述

NordHjem 采用测试金字塔来平衡反馈速度、维护成本与风险覆盖。目标是通过 **大量单元测试 + 中量集成测试 + 少量 E2E 测试**，在保证交付速度的同时降低回归风险。

本文档聚焦于可执行标准：

- 明确各层测试的职责边界（负责什么 / 不负责什么）
- 提供覆盖率目标路线图（按阶段推进）
- 统一 CI 集成和命名规范，确保团队执行一致

## 2. 测试金字塔定义

```text
        /  E2E  \         ← 少量：关键用户流程（Playwright）
       /----------\
      / Integration \     ← 中量：API 端点 + 数据库（node:test）
     /----------------\
    /   Unit Tests     \  ← 大量：独立函数和工具（Jest）
   /--------------------\
```

## 3. 各层职责明确定义

| 层级 | 工具 | 负责什么 | 不负责什么 | 运行时机 |
|------|------|----------|-----------|---------|
| Unit Tests | Jest (后端 + 前端) | 纯函数、工具方法、业务逻辑、组件渲染 | 数据库交互、API 调用、部署环境 | 每次 PR + push |
| Integration Tests | node:test (后端) | API 端点正确性、数据库 CRUD、服务间交互 | UI 渲染、样式 | 手动触发 / Nightly |
| E2E Tests | Playwright (前端) | 完整用户流程（浏览、购物车、结账） | 单个函数、内部逻辑 | 定时监控 + 部署后 |

## 4. 覆盖率目标路线图

| 阶段 | 时间 | 后端覆盖率 | 前端覆盖率 | 重点 |
|------|------|-----------|-----------|------|
| Phase 1（当前） | 2026 Q1 | 50% | 50% | 建立基线，跑通框架 |
| Phase 2 | 2026 Q2 | 65% | 65% | 补全核心模块测试 |
| Phase 3 | 2026 Q3 | 80% | 75% | 全面覆盖，高信心部署 |

## 5. CI 集成方式

- 后端 `ci.yml` → `unit-test` job → `npm run test:coverage`
- 前端 `ci.yml` → `unit-test` job → `yarn test:coverage`
- 覆盖率阈值在 `jest.config.ts` 中配置（`coverageThreshold`）
- 覆盖率低于阈值 → CI 失败 → PR 无法合并

## 6. 测试文件命名规范

- 单元测试：`*.unit.test.ts` 或放在 `__tests__/` 目录
- 集成测试：`*.test.ts`（在 `tests/` 目录）
- E2E 测试：`*.spec.ts`（在 `e2e-monitor/tests/`）

## 7. 编写测试的原则

1. **测试独立性**：每个测试可以单独运行，不依赖其他测试的执行顺序
2. **测试可读性**：测试名称用中文描述期望行为
3. **快速反馈**：单元测试总耗时应 < 60 秒
4. **Mock 原则**：外部依赖（API、数据库）在单元测试中一律 mock

## 8. 现状与执行补充

- 后端集成测试：当前已有 14 个 `node:test` 用例，作为 API + 数据库回归基线。
- 后端单元测试：已引入 Jest，用于扩大业务逻辑与工具层覆盖。
- 前端测试协同：E2E 由 `e2e-monitor/`（Playwright）承担，单元测试由 Jest 层补强。

执行建议：

- 新增后端模块时，默认先补 Unit Tests，再补对应 Integration Tests。
- 关键业务链路变更（购物车、结账、订单状态）需同步更新或新增 Playwright 场景。
- PR 描述中透明披露测试结果（通过项、失败项、未执行项及原因）。
