# MIGRATION-GUIDE

## 迁移背景

NordHjem 的迁移场景主要包括：

- **版本升级**：Medusa v2 patch 升级（安全修复、性能改进）。
- **环境迁移**：Railway 区域切换或服务资源升级。
- **数据库迁移**：schema 变更、索引调整、字段重命名与数据回填。

## 迁移步骤

通用流程（适用于版本、环境、数据库迁移）：

1. **备份**
   - 执行数据库快照与关键配置备份。
2. **测试迁移（staging）**
   - 在 staging 完整演练，记录耗时、失败点、回滚路径。
3. **维护窗口通知**
   - 提前通知 Owner 与相关干系人，明确影响范围与时间。
4. **执行迁移**
   - 按 runbook 分步骤执行，关键节点做状态确认。
5. **验证**
   - 核心 API、下单支付、Admin 操作、监控指标全面复检。
6. **恢复服务**
   - 确认无异常后恢复流量并持续观察。

## 验证方法

- **数据完整性检查**：迁移前后关键表行数对比、抽样核对订单与库存数据。
- **功能回归测试**：执行 smoke 测试 + 人工关键链路验证。
- **性能基准对比**：核验迁移前后 P95、错误率、资源占用是否在可接受范围。

## 风险与应急

| 风险 | 概率 | 影响 | 应对措施 |
| --- | --- | --- | --- |
| 迁移脚本执行超时 | 中 | 中-高 | 分批迁移 + 维护窗口预留 + 超时自动中止 |
| schema 变更导致应用不兼容 | 低-中 | 高 | 先灰度部署兼容版本，再执行最终切换 |
| 数据回填错误 | 低 | 高 | 先在 staging 验证回填脚本，生产执行前做只读演练 |
| 第三方回调（Stripe）堆积 | 中 | 中 | 迁移窗口临时限流，迁移后补处理并对账 |

**应急原则**：若迁移关键检查失败，立即停止并回滚到最近可用备份版本。
