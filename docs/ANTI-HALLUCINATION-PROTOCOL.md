# Anti-Hallucination Protocol (AHP)

> 全 AI 团队的核心风控机制
> 本协议具有最高优先级，所有角色必须遵守

---

## 核心原则

**AI 团队最大的风险不是效率，是幻觉。**

AI 可能：
- 编造不存在的 API 或库函数
- 基于错误假设做出看似合理的技术决策
- 自信地给出错误的信息
- 在不确定时不说"我不确定"

**对策：多维度审核，不同视角交叉验证。**

---

## 审核维度定义

| 维度 | 视角 | 检查什么 | 由谁负责 |
|------|------|---------|---------|
| **代码正确性** | 这段代码能跑吗？ | 语法、逻辑、运行时行为 | Domain Lead |
| **架构一致性** | 这个设计放在系统里合理吗？ | 模式一致、模块边界、技术债务 | CTO |
| **事实准确性** | 技术假设是真的吗？ | 官方文档验证、版本兼容性、API 实际行为 | Perplexity（外部调研） |
| **边界与失败** | 什么情况下会出错？ | 边界值、并发、网络失败、数据异常 | QA Lead |
| **安全合规** | 有没有安全漏洞？ | 注入、越权、数据泄露、依赖漏洞 | CI 自动化 + DevOps Lead |
| **用户价值** | 这真的是用户需要的吗？ | 需求匹配度、用户体验、验收标准 | Product Manager |

---

## 何时触发多维度审核

### Level 1: 标准审核（单维度）

适用于：日常开发、文档更新、小 bug 修复

```
Domain Lead review → merge
```

### Level 2: 增强审核（双维度）

适用于：新功能、API 变更、数据库变更

```
Domain Lead review（代码正确性）
+ CTO review（架构一致性）
→ merge
```

### Level 3: 完整审核（多维度）

适用于：架构决策(ADR)、新技术引入、生产部署、安全相关变更

```
Domain Lead review（代码正确性）
+ CTO review（架构一致性）
+ Perplexity 调研报告（事实准确性）
+ QA 测试报告（边界与失败）
→ CTO 综合判断 → merge
```

### Level 4: 最高审核（全维度 + Owner）

适用于：重大方向决策、技术栈变更、数据迁移、安全事件

```
全部 Level 3
+ Product Manager 验证（用户价值）
+ Owner 最终决策
→ 执行
```

## Perplexity 外部调研协议

Perplexity Max 是团队的"外部专家顾问"，负责：

### 调研场景
- 验证技术假设（"Medusa v2 真的支持这个模式吗？"）
- 技术选型对比（"Redis vs Memcached 在我们场景下的对比"）
- 最佳实践调研（"生产环境 Next.js 的 Sentry 配置最佳实践"）
- 安全漏洞验证（"这个 CVE 真的影响我们的版本吗？"）
- 竞品分析（产品经理需求）

### 调研流程
```
1. CTO 或负责人明确调研问题（具体、可验证）
2. 通过 Owner 提交给 Perplexity（Owner 中转）
3. Perplexity 输出调研报告
4. CTO 审核报告，提取可执行结论
5. 结论写入 Issue 或 ADR，形成决策依据
```

### 调研要求
- 问题必须具体："Medusa v2.13.x 的 workflow API 是否支持 parallel execution" ✅
- 不能模糊："Medusa 怎么样" ❌
- 报告必须包含信息来源（官方文档、GitHub Issue、Stack Overflow 链接）
- 如果 Perplexity 说"不确定"，这本身就是有价值的信号

## AI 幻觉高风险场景

以下场景特别容易出现幻觉，**必须触发 Level 3+ 审核**：

| 场景 | 风险 | 对策 |
|------|------|------|
| 引用第三方库 API | AI 可能编造不存在的方法 | Perplexity 查官方文档确认 |
| 版本兼容性判断 | AI 可能记错版本号 | 查 package.json + 官方 changelog |
| 性能数据 | AI 可能编造 benchmark 数据 | 实际运行测试验证 |
| 安全漏洞评估 | AI 可能低估或高估风险 | CVE 数据库 + Perplexity 交叉验证 |
| 配置参数 | AI 可能给出过时的配置 | 查最新官方文档 |
| 云服务 pricing | AI 可能使用过期价格 | Perplexity 实时查询 |

## CTO 把控原则

作为 CTO，我负责判断每个决策需要哪个审核 Level：

**升级审核 Level 的信号：**
- 影响范围超过 1 个模块
- 不可逆的变更（数据迁移、API 废弃）
- 涉及金钱（支付、订单）
- 涉及用户数据（隐私、安全）
- 我自己不确定的技术判断
- 团队中有人说"我觉得应该可以"但没有确凿证据

**降级审核 Level 的条件：**
- 纯文档变更
- CI 配置微调
- 已有先例的重复模式

**铁律：宁可多审一层，不可漏审一层。审核的成本是分钟，幻觉的代价是天。**

---

## 每个角色的防幻觉责任

| 角色 | 责任 |
|------|------|
| **所有人** | 不确定时说"我不确定"，而不是猜测 |
| **所有人** | 引用第三方时附上来源，没有来源就标注"需验证" |
| **CTO** | 判断审核 Level，协调多维度审核 |
| **Domain Leads** | 在自己领域内识别幻觉风险点 |
| **QA Lead** | 通过测试验证假设，而不是相信假设 |
| **Perplexity** | 提供有来源的事实验证 |

---

> 本协议由 CTO 维护。任何角色发现可能的幻觉风险，必须立即标记并上报。
> "我不确定"是最有价值的四个字。
